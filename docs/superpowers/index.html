Stakeholder Brief

Email Migrator — Phase 1

Systematic email template migration at scale.

SourceSystem → Canonical Taxonomy → TargetSystem

We built a tool that migrates email templates from SourceSystem to TargetSystem systematically, safely, and at scale. Instead of manually rewriting each template, the tool reads each source template, classifies every content and layout element against a shared taxonomy, maps those elements to their TargetSystem equivalents, and produces a structured migration spec. No target output is generated until a human reviewer has approved the spec.

Phase 1 is complete: the full pipeline runs, produces reviewed-ready migration specs for any number of HTML templates, and includes a local browser-based review UI.

The Problem

We have hundreds of email templates. Manual migration is risky.

  • Templates contain personalisation variables, compliance disclaimers, legal language
  • Manual copy-paste misses variables, breaks links, and loses context
  • Different people migrating different templates produce inconsistent results
  • No audit trail: how do you know what was changed and why?
  • Scale: hundreds of templates × multiple reviewers = months of error-prone work
The cost of getting this wrong is high. A broken transactional email or missing disclaimer is a customer experience failure, or worse.

Our Approach: A Translation Layer

We don't copy templates. We translate them through a shared language.

SourceSystem template
        ↓
  Canonical Taxonomy   ← shared vocabulary, defined once
        ↓
TargetSystem mapping
        ↓
  Human review gate
        ↓
TargetSystem output
  • Every element in every source template is classified against the canonical taxonomy
  • The taxonomy is the contract between the two systems
  • Mappings are rule-based, deterministic, and auditable
  • No output is produced without human sign-off
This is safer than a migration script and faster than manual rewriting.

The Canonical Taxonomy

A shared vocabulary for everything in an email template.

LayerWhat it classifiesExamples
Content blocksWhat the text isheadline, preheader, body text, CTA, disclaimer, footer
UI modulesHow it's laid outheader, hero, text block, button, divider, footer

Every block and module gets:

  • A stable ID: payment-reminder:headline:0
  • A type: one of the canonical vocabulary values
  • Variables extracted: {{firstName}}, {{amount}}, {{dueDate}}
  • A mapping result: which TargetSystem component it maps to, with a confidence score

How the Pipeline Works

INGEST     → Read HTML files, derive stable template IDs
EXTRACT    → Parse each template: content blocks, UI modules, variables
NORMALIZE  → Assign stable canonical IDs, validate structure
MAP        → Match each component to a TargetSystem equivalent
ASSESS     → Run QA rules, score confidence, write spec to disk ✎
REVIEW     → Human approves or flags in local browser UI
EXPORT     → Generate TargetSystem output (Phase 2)
  • Stages 1–5 run automatically from a single CLI command
  • Nothing is written to disk until ASSESS
  • REVIEW requires human action before EXPORT can proceed
  • EXPORT is hard-gated: it only runs after full approval

Human Review is Mandatory

The system will not generate target output without a human approving the migration spec.

StatusMeaningWhat happens
ready All components at 100% confidence, no QA issues Available for batch approval
needs_review One or more partial matches or QA warnings Must be reviewed individually
blocked QA errors (empty template, missing compliance) Pipeline exits non-zero. Cannot proceed.

Stakeholder

Why This Matters

Scale without sacrifice

The tool can process hundreds of templates in seconds. Every template gets the same rigorous classification regardless of who runs it or when.

Variables never get lost

All personalisation tokens ({{firstName}}, {{amount}}, etc.) are extracted automatically and validated bidirectionally — every variable in a content block is also declared at the template level, and every declared variable must appear somewhere in the content.

Compliance is tracked, not assumed

The QA engine checks for required compliance markers. A template that's missing a required disclaimer will be flagged blocked before it gets anywhere near a reviewer.

Every decision is auditable

The migration spec JSON is the permanent record. It captures what was mapped, what confidence score it got, why, who reviewed it, and when. This is not possible with manual migration.

Phase 2 is uncoupled from Phase 1

When the TargetSystem export adapter is built, it works from the already-reviewed specs. There is no need to re-review templates or re-run the pipeline just because the export format changed.

User Guide

What inputs it expects

A directory of .html or .htm email template files. File names become template IDs (payment-reminder.html → payment-reminder). Names must be unique after normalisation.

# Valid
payment-reminder.html
welcome-email.html
account-statement.html

# Will error — same slug
Payment Reminder.html   ← clashes with payment-reminder.html

What outputs it generates

One JSON migration spec per template, written to --specs. Each spec contains: template_id, content_blocks, ui_modules, variables, mapping_results, status, review_notes, assessed_at.

How to run the pipeline

npx tsx src/cli/index.ts run \
  --source ./templates \
  --specs ./specs

On completion you'll see a summary:

--- Pipeline complete ---
Total:        3
Ready:        2
Needs review: 1
Blocked:      0

✓ Specs written to ./specs

If blocked templates exist, the process exits non-zero:

✗ 1 template(s) are BLOCKED. Review spec files in ./specs for error details.

How to launch review

npx tsx src/cli/index.ts review \
  --specs ./specs \
  --source ./templates \
  --port 3333

How approvals work

  1. Sidebar lists all templates with status badges
  2. Click a template to open the full spec detail
  3. Review content blocks, variables, mapping results, QA notes
  4. Click Approve or Flag (prompts for a note)
  5. Batch approve approves all ready templates at once
Blocked templates cannot be approved. Fix the source, re-run, and review again.

Common workflow

1. Copy .html templates into a directory
2. npx tsx src/cli/index.ts run --source ./templates --specs ./specs
3. If blocked → fix source, re-run
4. npx tsx src/cli/index.ts review --specs ./specs --source ./templates
5. Batch-approve "ready" · Review "needs_review" individually
6. Export (Phase 2)

Known Issues

All items below are non-blocking Phase 1 hardening items. The pipeline runs correctly, all 57 tests pass, and neither item affects normal use of the tool in its current local-only deployment context.
KI-001

saveSpec() write path lacks path-safety guard

Severity: Minor hardening Component: src/review/server.ts Affects: approve & flag endpoints

The read paths use a safeSpecPath() helper to prevent path traversal. The saveSpec() write path constructs its path from template.template_id in the JSON contents without the same guard. In theory, a crafted spec file with template_id: "../../etc/shadow" could write outside specsDir.

Why it's low risk: Server is local-only. Exploiting this requires filesystem write access to specsDir. In normal use, all IDs come from slugify() which strips everything outside [a-z0-9-].

Intended fix: Apply the same safeSpecPath() guard in saveSpec(), or validate template_id matches [a-z0-9-]+ before constructing the path.

KI-002

One onclick in renderSidebar passes template_id without esc()

Severity: Minor hardening Component: src/review/ui/app.html Affects: sidebar rendering

The review UI applies esc() throughout to prevent XSS. One onclick in renderSidebar passes t.template_id directly into a JS string literal inside an HTML attribute without going through esc().

Why it's low risk: template_id values always come through slugify(), which strips all characters outside [a-z0-9-]. Injection requires manually writing a crafted spec file to disk.

Intended fix: Wrap t.template_id in esc() at the affected onclick in renderSidebar.
